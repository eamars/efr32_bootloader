74 *
75 * Note(s) : (1) RTOS_ERR_CFG_LEGACY_EN is provided to allow usage of existing code without needing to
76 * modify anything regarding error codes. New applications should either completely remove
77 * this configuration define or set it to DEF_DISABLED, to allow usage of extended errors
78 * and other advantages such as unified error codes.
79 *
80 * (2) RTOS_ERR_CFG_EXT_EN allows to configure whether the RTOS_ERR type contains only an error
81 * code (DEF_DISABLED) or contains more debug information (DEF_ENABLED). If set to
82 * DEF_ENABLED, a string containing the file name and line number where the error has been
83 * set and also the function name, if compiling in C99, will be included in the RTOS_ERR
84 * type. Setting this configuration to DEF_ENABLED may have an impact on performance and
85 * resource usage, it is recommended to set it to DEF_DISABLED once development is
86 * complete. This feature cannot be used if RTOS_ERR_CFG_LEGACY_EN is set to DEF_ENABLED.
87 *
88 * (3) RTOS_ERR_CFG_STR_EN allows to have strings associated with each error code, in order to
89 * print them. If set to DEF_DISABLED, the error code enum value will be outputted instead.
90 * For example, if set to DEF_ENABLED, it would be possible to print RTOS_ERR_NONE or
91 * RTOS_ERR_INVALID_ARG as a string instead of printing the numerical value associated,
92 * which would be 0 for RTOS_ERR_NONE and higher than 0 for RTOS_ERR_INVALID_ARG.
93 *********************************************************************************************************
94 *********************************************************************************************************
95 */
