3 // Hashing a file can be expensive for small processors. The maxHashCalculations
4 // determines how many iterations are made before returning back to the caller.
5 // A value of 0 indicates to completely calculate the digest before returning.
6 // A value greater than 0 means that a number of hashes will be performed and
7 // then the routine will return EMBER_AF_IMAGE_VERIFY_IN_PROGRESS. In that
8 // case it is expected that this function must be called repeatedly until it
9 // returns another status code.
10 // When EMBER_AF_IMAGE_VERIFY_WAIT is returned then no further calls
11 // are necessary. The verify code will fire the callback
12 // emAfOtaVerifyStoredDataFinish() when it is has a result.
13 EmberAfImageVerifyStatus emAfOtaImageSignatureVerify(uint16_t maxHashCalculations,
14  const EmberAfOtaImageId* id,
15  bool newVerification);

17 // This is the maximum number of digest calculations we perform per call to
18 // emAfOtaImageSignatureVerify(). Arbitrary chosen value to limit
19 // time spent in this routine. A value of 0 means we will NOT return
20 // until we are completely done with our digest calculations.
21 // Empirically waiting until digest calculations are complete can
22 // take quite a while for EZSP hosts (~40 seconds for a UART connected host).
23 // So we want to make sure that other parts of the framework can run during
24 // this time. On SOC systems a similar problem occurs. If we set this to 0
25 // then emberTick() will not fire and therefore the watchdog timer will not be
26 // serviced.
27 #define MAX_DIGEST_CALCULATIONS_PER_CALL 5
