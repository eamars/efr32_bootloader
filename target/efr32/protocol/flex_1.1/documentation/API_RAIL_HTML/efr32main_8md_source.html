1 EFR32 {#efr32_main}
2 ============
3 
4 While RAIL attempts to be chip agnostic there are certain hardware differences
5 that can't be overlooked. Where possible, missing features will be simulated in
6 software, but performance characteristics may vary. This section will cover what
7 differs for the EFR32 series of chips as well as any hardware specific
8 configurations or calibrations.
9 
10 ## Packet Buffer Restrictions
11 
12 Currently the RAIL library for the EFR32 allocates internal buffers to store the
13 receive and transmit data contiguously. These buffers are set to 512 bytes each
14 at build time and cannot be removed. This choice limits the maximum size of
15 receive and transmit packets in packet mode and determines the size of the FIFO
16 in FIFO mode. Since each receive packet has 8 bytes of overhead this means that
17 you can only receive up to one 504 byte packet. Transmit does not have this
18 restriction and you may send a full 512 byte packet.
19 
20 In the future these buffers will likely be made user configurable to accommodate
21 different applications. Note, however, that the EFR32 will always require a
22 power of 2 sized buffer due to the underlying hardware.

24 ## Data Reception Sources
25 
26 When receiving data, you can configure hardware to provide data from three different hardware sources.
27 
28 First, you can configure the hardware to provide a packet of information.
29 This configuration utilizes the built-in demodulator and frame controller.
30 Use \ref RAIL_RxDataSource_t::RX_PACKET_DATA to enable this receive source.
31 
32 Second, you can configure the hardware to provide data directly from the
33 demodulator. In this mode, the hardware demodulator is used, but the user is
34 responsible for implementing frame controller functionality. (i.e. Preamble
35 detection, sync word detection, CRC validation, etc. must be performed in
36 software by the application.) All data returned is represented as 8-bit
37 2's-compliment values. Use \ref RAIL_RxDataSource_t::RX_DEMOD_DATA to enable
38 this receive source.
39 
40 Third, you can configure the hardware to provide data directly from the I and
41 Q ADCs. In this mode, the user is responsible for implementing demodulator and
42 frame controller functionality. The receive signal hardware has a 19-bit
43 dynamic range, and the user can select whether to return the upper 16 bits of
44 the 19-bit value (\ref RAIL_RxDataSource_t::RX_IQDATA_FILTMSB) or the lower 16
45 bits (\ref RAIL_RxDataSource_t::RX_IQDATA_FILTLSB). All data returned is
46 represented as 16-bit 2s-compliment values. The I and Q values are put into the
47 buffer in an alternating fashion where the first 16-bit value is the first I
48 ADC value, and the second 16-bit value is the first Q ADC value, etc.

50 ## Hardware Initialization
51 
52 The EFR32 comes with versions of EMLIB and EMDRV to create a basic HAL layer.
53 A lot of this initialization code is completely up to you, but there are a
54 couple of requirements when you're building a RAIL app. Specifically, the radio
55 will only work if you are running off a high precision crystal oscillator. Since
56 some APIs will assume this is running you should make sure to initialize and
57 switch to the crystal before calling any radio APIs.
58 
59 For the WSTK you can use the crystal configuration stored in the bspconfig.h
60 header file for your specific kit. Example code for this is shown below. If you
61 have a custom hardware layout you may want to create your own HFXOInit structure
62 to account for things like your specific CTUNE value.
63 
64 ~~~~~~~~~~{.c}
65 #include "bsp.h" // Contains WSTK versions of the HFXO init structure
66 
67 void efrStartup(void)
68 {
69  CMU_HFXOInit_TypeDef hfxoInit = CMU_HFXOINIT_WSTK_DEFAULT;
70 
71  // Initialize the HFXO using the settings from the WSTK bspconfig.h
72  // Note: This configures things like the capacitive tuning CTUNE variable
73  // which can vary based on your hardware design.
74  CMU_HFXOInit(&hfxoInit);
75 
76  // Switch HFCLK to HFXO and disable HFRCO
77  CMU_ClockSelectSet(cmuClock_HF, cmuSelect_HFXO);
78  CMU_OscillatorEnable(cmuOsc_HFRCO, false, false);
79 }
80 ~~~~~~~~~~

82 ### Interrupt Vectors
83 
84 The RAIL library for EFR32 implements all of the interrupt vectors for radio
85 peripherals. This is required for the RAIL library to function correctly. RAIL
86 does not, however, set the priorities of these interrupt vectors. That must be
87 handled by your application using the CMSIS NVIC_SetPriority() API or direct
88 manipulation of the NVIC. Below is the full list of interrupts used by the
89 radio. We recommend running them all at the same priority which is what our RAIL
90 applications do by default. You are free to choose what that priority is based
91 on the requirements of your application (e.g. putting them below FreeRTOS atomic
92 for access to OS functions in RAIL callbacks). Keep in mind that putting the
93 radio interrupts at too low a priority can cause missed packets and other radio
94 performance issues. A restriction on EFR32xG12/13 is not to disable radio
95 interrupts for longer than quarter of the RAIL timebase, which is 2^32/4
96 microseconds or 18 minutes. Please see \ref EFR32xG1x_Interrupts or follow
97 the links below for more information.

112 ## Radio Specific Initialization
113 
114 Currently there are two initialization functions for the radio that depend on
115 your board configuration and must be manually called at startup. You __MUST__
116 initialize the power amplifier (PA) in order to transmit. You __MUST__ also
117 initialize the PA before initializing RAIL for the library to properly account
118 for the PA settings during timing calculations. You may initialize the packet
119 trace interface (PTI) if you want to use this for debugging.
120 
121 ### Power Amplifier (PA) Initialization
122 
123 There are several different PAs available for the EFR32 family of chips. The
124 specific set of PAs you have available and the supported power levels are
125 determined by what part number you're using. To fully understand your part and
126 what options you have please consult the data sheet for your part.
127 
128 Once you have chosen the PA configuration that you need it's important that you
129 initialize the PA before doing any radio operations, but after the HFXO has been
130 enabled and switched to. The structures and APIs for doing this can be found in
131 the pa.h file. An example of initializing the high power 2.4GHz PA is shown
132 below.
133 
134 ~~~~~~~~~~{.c}
135 #include "pa.h"
136 
137 void initPa(void)
138 {
139  RADIO_PAInit_t paInit = {
140  PA_SEL_2P4_HP, // Mode: PA_SEL_SUBGIG, PA_SEL_2P4_LP, PA_SEL_2P4_HP
141  PA_VOLTMODE_DCDC, // VPA mode: PA_VOLTMODE_VBAT, PA_VOLTMODE_DCDC
142  100, // Desired output power in dBm * 10
143  0, // Output power offset in dBm * 10
144  10, // Desired ramp time in us
145  };
146 
147  // Initialize the PA (must be done only after the HFXO is enabled)
148  if (!RADIO_PA_Init(&paInit)) {
149  // Error: The PA could not be initialized due to an improper configuration.
150  // Please ensure your configuration is valid for the selected part.
151  while(1);
152  }
153 }
154 ~~~~~~~~~~

156 ### Packet Trace Interface (PTI) Initialization
157 
158 Packet trace on the EFR32 provides a mechanism for viewing transmitted and
159 received radio packets for network sniffing or debugging. It can also be
160 captured by a WSTK and sent to Simplicity Studio for viewing the data in its
161 Network Analyzer tool.
162 
163 If you want to use this functionality, you must configure what pins you want to
164 use for this output and, optionally how you want to format the data. Note that
165 the WSTK requires the following output format:
166 
167  Mode: 8 bit UART mode
168  Baudrate: 1.6Mbps
169  Framing Signal: Enabled
170 
171 If you change anything here it is currently unsupported by the WSTK. In the
172 future we may include support for alternate output modes.
173 
174 To choose the pins you want you must look in the data sheet for your part and
175 select a valid route location for the FRC_DOUT and FRC_DFRAME signals. These map
176 to PTI.DATA and PTI.FRAME on the WSTK respectively. Once you've found pins that
177 work for your hardware you need to configure the RouteLocation, port, and pin
178 fields in the PTI initialization structure. When using the WSTK for example,
179 you should initialize things as follows.
180 
181 ~~~~~~~~~~{.c}
182 #include "pti.h"
183 
184 void initPti(void)
185 {
186  RADIO_PTIInit_t ptiInit = {
187  RADIO_PTI_MODE_UART, // Only supported output mode for the WSTK
188  1600000, // Choose 1.6 Mbps for the WSTK
189  6, // WSTK uses location 6 for DOUT
190  gpioPortB, // FRC_DOUT#6 is PB12
191  12, // FRC_DOUT#6 is PB12
192  6, // UNUSED IN UART MODE
193  gpioPortB, // UNUSED IN UART MODE
194  11, // UNUSED IN UART MODE
195  6, // WSTK uses location 6 for DFRAME
196  gpioPortB, // FRC_DOUT#6 is PB13
197  13, // FRC_DOUT#6 is PB13
198  };
199 
200  // Initialize the Packet Trace Interface (PTI)
201  RADIO_PTI_Init(&ptiInit);
202 }

205 ### Other Radio GPIO Functions
206 
207 There are various useful signals related to the radio that can be output on a
208 GPIO using the Peripheral Reflex System (PRS). The PRS is an advanced system
209 where you can route signals to channels and then output those channels to a
210 number of configurable locations. There's a lot more to it than that, so
211 consult the reference manual chapter on the PRS if you want to understand it
212 better.
213 
214 A list of some of the most interesting PRS signals related to the radio is shown
215 below along with how to enable them in the PRS.
216 
217 
218 | Signal | Summary | PRS Source (efr32xg1) | PRS Source (efr32xg12) | PRS Signal |
219 |:----------------|:----------------------|:----------------------|:----------------------|:-----------|
220 | RACACTIVE | Radio enabled | 0x20 | 0x51 | 0x00 |
221 | RACTX | Transmit mode enabled | 0x20 | 0x51 | 0x01 |
222 | RACRX | Receive mode enabled | 0x20 | 0x51 | 0x02 |
223 | RACLNAEN | LNA enabled for Rx | 0x20 | 0x51 | 0x03 |
224 | RACPAEN | PA enabled for Tx | 0x20 | 0x51 | 0x04 |
225 | MODEM_FRAMEDET | Frame detected | 0x26 | 0x56 | 0x00 |
226 | MODEM_PREDET | Preamble detected | 0x26 | 0x56 | 0x01 |
227 | MODEM_TIMDET | Timing detected | 0x26 | 0x56 | 0x02 |
228 | MODEM_FRAMESENT | Frame sent | 0x26 | 0x56 | 0x03 |
229 | MODEM_SYNCSENT | Syncword sent | 0x26 | 0x56 | 0x04 |
230 | MODEM_PRESENT | Preamble sent | 0x26 | 0x56 | 0x05 |
231 
232 
233 The example below shows you how to configure a PRS channel to output RACRX on
234 a GPIO. For this example we assume you're using the SLWSTK6001A_EFR32MG kit. The
235 specified configuration will put RACRX on PRS Channel 0 and output PRS Channel 0
236 on the EFR32's PC10 which is wired to WSTK_P12 and EXP_HEADER15 on the WSTK.
237 
238 ~~~~~~~~~{.c}
239 #include "em_cmu.h"
240 #include "em_gpio.h"
241 #include "em_device.h"
242 
243 void enableDebugGpios(void)
244 {
245  // Turn on the PRS and GPIO clocks so we can access their registers
246  CMU_ClockEnable(cmuClock_PRS, true);
247  CMU_ClockEnable(cmuClock_GPIO, true);
248 
249  // Configure PC10 as an output
250  GPIO_PinModeSet(gpioPortC, 10, gpioModePushPull, 0);
251 
252  // Configure PRS Channel 0 to output RACRX
253  PRS->CH[0].CTRL = (0x20 << _PRS_CH_CTRL_SOURCESEL_SHIFT)
254  | (0x02 << _PRS_CH_CTRL_SIGSEL_SHIFT)
255  | PRS_CH_CTRL_EDSEL_OFF;
256 
257  // Configure PRS Channel 0 to use output location 12 (PC10 - see datasheet)
258  PRS->ROUTELOC0 &= ~_PRS_ROUTELOC0_CH0LOC_MASK;
259  PRS->ROUTELOC0 |= PRS_ROUTELOC0_CH0LOC_LOC12;
260 
261  // Enable PRS Channel 0
262  PRS->ROUTEPEN |= PRS_ROUTEPEN_CH0PEN;
263 }
264 ~~~~~~~~~

270 library, EMLIB, and CMSIS. Below is a complete list of these dependencies. If

297 ## Entropy Generation
298 
299 The EFR32 supports true entropy collection using the radio. It is able to
300 collect 1 bit per radio clock cycle while in receive mode. This means that we
301 must enable the receiver when collecting entropy and if you attempt to transmit
302 or otherwise delay entry into receive it will take us a longer time to collect
303 the data. You can still receive packets while collecting entropy and this does
304 mean that requesting random numbers longer than the length of your preamble and
305 sync word may trigger a packet reception. This may be fixed in a future revision
306 but for now this is done to ensure proper packet reception during random data
307 collection.
308 
309 Due to the nature of random data collection we will always wait for the full
310 amount requested so we should return either the number of bytes requested or
311 zero. Zero is returned if the radio is uninitialized and cannot be enabled for
312 collection.

314 ## Radio Calibration
315 
316 The EFR32 supports two radio calibration options: image rejection (IR)
317 calibration and VCO temperature calibration. You may choose to enable one or both
318 of these based upon your specific situation. We always recommend using both, but
319 here is more information.
320 
321 * Image Rejection Calibration (IRCAL)
322  * This calibration should be run once every time your PHY config changes. It
323  is based on the modulation scheme, frequency band, and other radio settings.
324  RAIL will request this via the RAILCb_CalNeeded() callback with the
325  \ref RAIL_CAL_ONETIME_IRCAL bit set in RAIL_CalPendingGet().
326  * Using a proper value for this will improve sensitivity by several dBm, so
327  it's highly recommended.
328  * It can take on the order of 700 ms to complete this calibration.
329  You may want to save off a known good value for this calibration and load
330  it each time you switch PHYs to save time.
331  * This calibration should be initialized before channel configuration in order
332  to provide parameters specific to the current radio config. The
333  initialization parameters come from Simplicity Studio's application builder.
334  * You cannot use the radio while this calibration is being performed or you may
335  generate an incorrect calibration. Application code should ensure that the
336  radio remains in the idle state during this calibration.
337  * This calibration should be performed only after the radio config has been
338  downloaded to the radio, calibration initialization has occurred, and the
339  frequency channel has been configured.
340 * VCO Temperature Calibration (VCO_TEMPCAL)
341  * If you sit in receive for a very long time and experience temperature swings
342  it's possible for the radio to drift off frequency. When this situation is
343  detected RAIL will request this calibration via the RAILCb_CalNeeded()
344  callback with the \ref RAIL_CAL_TEMP_VCO bit set in RAIL_CalPendingGet().
345  * This calibration is automatically run every time we enter receive. So if you
346  are frequently forcing a re-enter into receive mode you may not need to
347  enable this calibration.
348  * On EFR, the application will get a RAIL_CalNeeded() callback for VCO
349  calibration when the absolute temperature crosses 0C degrees as well as
350  when the temperature delta from the last calibration increases or decreases
351  by 70C.
352  * We always recommend that you enable and handle this calibration since it
353  doesn't add much overhead and is much safer if you ever do end up in this
354  situation.
